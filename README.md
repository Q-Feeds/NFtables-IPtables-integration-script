## Q-Feeds NFtables/IPtables Integration

Script to implement Q-Feeds blocklists directly on `nftables` (this repository currently contains the `nftables` installer/update script).

This solution periodically downloads the latest threat intelligence feed from Q-Feeds and applies it as firewall rules, allowing you to:

- **Block incoming connections** from known malicious IPs
- **Block outgoing connections** to known malicious IPs
- **Whitelist your own IPs/CIDRs** so you never lock yourself out, even if they appear in the feed

---

## Components and Files

- **`NFtables-installer.sh`**  
  One-time installer script that:
  - Prompts you for your Q-Feeds API token and options
  - Generates the configuration file at `/etc/qfeeds/qfeeds_config.conf`
  - Installs the main updater script at `/usr/local/bin/update_qfeeds_blocklist.sh`
  - Sets up a cron job to keep the blocklist up to date
  - Runs an initial update to validate that everything works

- **`/usr/local/bin/update_qfeeds_blocklist.sh`** (created by the installer)  
  The scheduled update script that:
  - Fetches the latest IP list from Q-Feeds
  - Updates the `nftables` sets used for blocking
  - Updates the optional whitelist sets
  - Saves the firewall rules for persistence (where supported)

- **`/etc/qfeeds/qfeeds_config.conf`**  
  Configuration file generated by the installer. You can edit it later if you want to change behavior without rerunning the installer.

## What the Installer Asks You

When you run:

```bash
sudo ./NFtables-installer.sh
```

you will be asked the following questions.

### 1) Q-Feeds API Token (required)

- **Prompt**:  
  `Enter your Q-Feeds API Token:`
- This token authenticates you against the Q-Feeds API.
- The installer **refuses to continue** if this is empty.

The value is stored in the config as:

```bash
API_TOKEN="your_token_here"
```

### 2) Feed Type (optional)

- **Prompt**:  
  `Enter feed type [default: malware_ip]:`
- Default is `malware_ip`.
- Only change this if Q-Feeds has provided you with a different feed type.

Stored as:

```bash
FEED_TYPE="malware_ip"
```

### 3) Limit of IPs to Fetch (optional)

- **Prompt**:  
  `Enter the limit of IPs to fetch (leave empty for no limit):`
- If you **press Enter**, no limit is applied (all entries from the feed are eligible to be pulled).
- If you enter a number (for example `50000`), that value is used to cap how many entries are pulled from the feed.

Stored as:

```bash
LIMIT=50000         # if you entered 50000
# or empty/omitted  # if you pressed Enter for "no limit"
```

### 4) Directional Blocking – Incoming / Outgoing

You can choose if Q-Feeds should only protect you from **incoming** malicious connections, or also block **outgoing** traffic to known-bad IPs.

- **Incoming (default: enabled)**  

  - **Prompt**:  
    `Block INCOMING connections from malicious IPs? [Y/n]:`

  - If you press **Enter** or type `y`/`yes`:  
    `BLOCK_INCOMING="yes"`  
    Incoming connections from blacklisted IPs will be blocked.

  - If you type `n`/`no`:  
    `BLOCK_INCOMING="no"`  
    No special Q-Feeds handling for incoming traffic will be created.

- **Outgoing (default: disabled)**  

  - **Prompt**:  
    `Block OUTGOING connections to malicious IPs? [y/N]:`

  - If you type `y`/`yes`:  
    `BLOCK_OUTGOING="yes"`  
    Outgoing traffic to listed IPs will be blocked.

  - If you press **Enter** or type `n`/`no`:  
    `BLOCK_OUTGOING="no"`  
    No special Q-Feeds handling for outgoing traffic will be created.

These options are stored in the config as:

```bash
BLOCK_INCOMING="yes"  # or "no"
BLOCK_OUTGOING="no"   # or "yes"
```

If the update script is run with an **older config** that doesn’t contain these keys, it assumes:

- `BLOCK_INCOMING="yes"`
- `BLOCK_OUTGOING="no"`

This means: by default, it behaves like a pure incoming blocklist (the same as the original script).

### 5) Whitelist (optional, to avoid locking yourself out)

You can define IPs or CIDR ranges that should **never be blocked**, even if they appear in the Q-Feeds list.

- **Prompt 1**:  
  `Configure a whitelist of IPs/CIDRs that must NEVER be blocked? [y/N]:`

  - `y`/`yes`: you will be asked for IPv4 and IPv6 whitelist entries.
  - `n` or Enter: no whitelist is configured.

- **Prompt 2 (IPv4)**:  

  `Enter IPv4 whitelist (comma-separated, e.g. 1.2.3.4,5.6.7.8) [leave empty for none]:`

  Examples:

  - `203.0.113.10`
  - `203.0.113.10,198.51.100.0/24`

  If you leave this blank, no IPv4 whitelisted addresses are set.

- **Prompt 3 (IPv6)**:  

  `Enter IPv6 whitelist (comma-separated, e.g. 2001:db8::1,2001:db8::2) [leave empty for none]:`

  Same idea as IPv4, but for IPv6.

These are stored as:

```bash
WHITELIST_V4="203.0.113.10,198.51.100.0/24"
WHITELIST_V6="2001:db8::1"
```

You can edit the config later to add/remove IPs if needed.

## What the Updater Script Actually Does

The main updater script `/usr/local/bin/update_qfeeds_blocklist.sh` is what runs on a schedule (via cron) and enforces the policy you chose during installation.

### 1) Loads the Configuration

It reads `/etc/qfeeds/qfeeds_config.conf` and uses:

- `API_TOKEN`, `FEED_TYPE`, `LIMIT`
- `BLOCK_INCOMING`, `BLOCK_OUTGOING`
- `WHITELIST_V4`, `WHITELIST_V6`

If some of the newer options are missing (for example if you installed an older version previously), it falls back to safe defaults:

```bash
BLOCK_INCOMING="yes"
BLOCK_OUTGOING="no"
```

### 2) Fetches the Q-Feeds Blocklist

It builds an API URL based on your config, then uses `curl` to fetch the IP list:

- If the HTTP status code is **not 200**, or the response is empty, it logs an error and exits.
- If the response looks valid, it copies the IPs into a temporary file for processing.

### 3) Manages nftables Tables, Sets, and Chains

The script manages its own `nftables` tables and sets, under:

- `table ip qfeeds`   (IPv4)
- `table ip6 qfeeds`  (IPv6)

Within those tables it maintains:

- **Blacklist sets** (used for blocking):

  ```bash
  qfeeds_blacklist_v4
  qfeeds_blacklist_v6
  ```

- **Whitelist sets** (used to always allow traffic before checking the blacklist):

  ```bash
  qfeeds_whitelist_v4
  qfeeds_whitelist_v6
  ```

Each run:

- Ensures the tables and sets exist.
- **Flushes** the blacklist sets before repopulating them from the latest Q-Feeds data.
- **Flushes** the whitelist sets before repopulating them from your configured whitelist IPs.

### 4) Incoming vs Outgoing Rules

Depending on your configuration:

- If `BLOCK_INCOMING="yes"`:

  - IPv4:
    - Ensures `chain ip qfeeds input-chain { type filter hook input priority 0; policy accept; }`
    - Ensures:

      ```nft
      ip s @qfeeds_whitelist_v4 accept
      ip s @qfeeds_blacklist_v4 drop
      ```

  - IPv6:
    - Ensures `chain ip6 qfeeds input-chain { type filter hook input priority 0; policy accept; }`
    - Ensures:

      ```nft
      ip6 s @qfeeds_whitelist_v6 accept
      ip6 s @qfeeds_blacklist_v6 drop
      ```

- If `BLOCK_OUTGOING="yes"`:

  - IPv4:
    - Ensures `chain ip qfeeds output-chain { type filter hook output priority 0; policy accept; }`
    - Ensures:

      ```nft
      ip d @qfeeds_whitelist_v4 accept
      ip d @qfeeds_blacklist_v4 drop
      ```

  - IPv6:
    - Ensures `chain ip6 qfeeds output-chain { type filter hook output priority 0; policy accept; }`
    - Ensures:

      ```nft
      ip6 d @qfeeds_whitelist_v6 accept
      ip6 d @qfeeds_blacklist_v6 drop
      ```

If you disable incoming or outgoing in the config, the corresponding chains/rules are not created/used.

> **Important:**  
> All of this is done within the dedicated `qfeeds` tables. Your existing firewall rules (for example in `table inet filter`) are not modified by this script.

### 5) Whitelist Population

For each run:

- Uses `WHITELIST_V4` and `WHITELIST_V6` (comma-separated IPs/CIDRs) from the config.
- Splits the lists, trims whitespace, and adds each entry to the corresponding whitelist set.
- Because the whitelist rules are always placed **before** the blacklist rules, these addresses are never blocked by Q-Feeds logic.

This is how you can safely add:

- Your own management IP(s)
- Critical partners or monitoring systems

without risking a lockout, even if those IPs ever appear in the threat feed.

### 6) Persisting the Rules

After updating the sets and rules, the script tries to persist the rules:

- If `netfilter-persistent` is available, it runs:

  ```bash
  netfilter-persistent save
  ```

- Otherwise, it logs a note suggesting that you save manually, usually with:

  ```bash
  nft list ruleset > /etc/nftables.conf
  ```

Persistence behavior depends on your Linux distribution and how your firewall is managed.

## Cron / Scheduling

The installer also configures a cron job:

- You are asked for a schedule (for example `*/20 * * * *`).
- The default is:

  ```text
  */20 * * * *
  ```

  which means “every 20 minutes”.

The cron entry runs:

```bash
/usr/local/bin/update_qfeeds_blocklist.sh
```

so your blocklist and whitelist logic stay automatically up to date without any manual action.

## Typical Customer Scenarios

- **“I only care about blocking bad IPs trying to reach my server.”**  
  - Answer **yes** to “Block INCOMING…?”, keep “Block OUTGOING…?” as **no** (default).
  - Optionally configure a whitelist with your own IP(s) so you never lock yourself out.

- **“I also want to stop my server from talking to known bad IPs.”**  
  - Answer **yes** to both incoming and outgoing blocking.
  - Add any essential external monitoring or partner IPs to the whitelist so their traffic is never interrupted.

- **“I want to make absolutely sure I can always reach the box.”**  
  - Enable a whitelist and enter your management IP(s) and any VPN/endpoints you use.
  - The whitelist is always checked **before** the blacklist, for both incoming and outgoing (if enabled).

If you are unsure about the best settings for your environment, start with:

- `BLOCK_INCOMING="yes"` (default)
- `BLOCK_OUTGOING="no"`
- A whitelist entry for your own IP(s)

You can then gradually tighten the policy later by editing `/etc/qfeeds/qfeeds_config.conf` and letting the cron job apply the new configuration.
